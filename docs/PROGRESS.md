## ✅ Done — Resit / fail handling (a failed job, redone; truthful net loss; DESIGN_SPEC §3, §6, §8)
- **Model**: a failed job is redone ON THE SAME `work_item` (owner's call: "both recorded against the job"). A controlled **reopen** moves work-state back (`delivered`/`confirmed` → `pending`, only via the resit path; the generic transition stays forward-only); money-state recomputes. The resit (second) writer is an **extra producer line + a leg** on the same job. The original writer's pay is reduced **auto by paid-status**: `reverseAmt = clamp(R, 0, max(0, outstanding))` posts a **negative reversing leg**, `chargeAmt = R − reverseAmt` posts an **`adjustment` clawback charge** — disjoint, so the same money is never both reversed and charged. Client side: default = resit billed 0 (original billing stands); escalation `zeroClientBilling` = a **negative client leg** (revenue → 0) + **void** the job's invoices + `recomputeMoneyState` → `unbilled`.
- **Loss is DERIVED, never stored**: `deriveJobPnl` (packages/shared/src/work.ts) over `job_pnl(work_item)` (revenue = Σ client legs; writer_cost = Σ legs to writer-typed parties **excluding the client→partner revenue handoff** so a multi-hat partner isn't miscounted; clawback = Σ `adjustment` charges) + the outcome's rework cost. `net = revenue − writerCost + clawback − reworkCost`; `isLoss = net < 0`. Surfaced on **GET /work/:id `.pnl`** (money-gated; `null` for a non-money caller) so a net loss is reported truthfully and never hidden. `deriveMargins` already shows the negative node; settlement pool goes negative.
- **Migration 0022**: `work_outcome.resit` flag; SECURITY DEFINER `party_earnings_outstanding` (owed−paid, drives the split) + `job_pnl` (the loss read model; admin isn't a leg/charge party). `POST /work/:id/resit` (`work:approve`) requires a recorded **`failed`** outcome (governance). Append-only throughout (reversing/new legs are negative/positive INSERTs, client-side id, no RETURNING); only legitimate UPDATEs are work-state, the outcome stamp, and invoice→void.
- **Web** (lean) `/resit` (admin, `work:approve`): pick a job → see its money-gated P&L (loss badge) → resit form (original writer + reduction, optional different resit writer, **re-bill to 0** toggle, rework cost). AppShell nav "Resit" gated `work:approve`.
- **Tests** (qa-test-writer; **no remaining bugs**): `packages/db/test/job-pnl.test.ts` (10) + `apps/api/test/resit-http.test.ts` (18) — two-writers-one-job, original-writer reduce (unpaid → reversing leg) / clawback (paid → `adjustment` due) / partial (reverseAmt+chargeAmt=R, no double-count), client-billed-0 (revenue→0, invoices void, money_state unbilled), loss reflected + pnl redaction, reopen, governance/authz (Writer 403; resit needs a failed outcome). DB 279/279 green; guard:no-stored-profit passes.
- **Reviews addressed**: qa found a real bug — invoice void used a non-existent `updated_at` column (fixed). security: **M2** writer_cost was tag-only → miscounted a multi-hat partner's revenue handoff (now excludes the client→partner leg); **M1** `originalWriterFromPartyId` required only when a reversing leg is posted; **M3** block `zeroClientBilling` when the client already paid (needs a reversing payment, not a void); **L1** reduction validated ≤ the writer's earning on this job. L2/precedent (`job_pnl`/`party_earnings_outstanding` rely on the money-gated/approve endpoint like `job_money`/`party_job_earnings`) logged in DECISIONS.
