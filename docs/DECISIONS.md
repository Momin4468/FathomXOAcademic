# DECISIONS.md

> Append-only architectural decision log. **Append (never edit/delete) an entry whenever a foundational decision is made** (per CLAUDE.md §1). Newest at the bottom. Format: date · decision · why. This is the "why things are the way they are" record — read before changing anything structural.

---

### 2026-06 · Stack: PostgreSQL + TypeScript + NestJS + Next.js + embedded Metabase
Postgres chosen over MySQL for native row-level security, JSONB (custom fields), and ledger/analytics strength — all core requirements. TS end-to-end for one-language maintainability. Modular monolith.

### 2026-06 · Profit is derived from legs, never stored
Margins/splits/commission computed at read time from the leg chain so they stay correct as date-versioned deal terms change. No profit/margin column anywhere.

### 2026-06 · Visibility enforced at the database (RLS), not just UI
A user may read a money leg only if SuperAdmin or their party is on it. Enforced by Postgres RLS so a UI bug cannot leak a price.

### 2026-06 · Personal Finance is a separable service, linked not merged
Own identity/data/subscription; one-way income API from business payouts; business cannot read into it; deactivating a brokerage account does not disable it.

### 2026-06 · Two parallel closes per job
Work-state (draft→pending→confirmed→delivered) and money-state (unbilled→invoiced→partial→settled) are independent.

### 2026-06-27 · Monorepo tooling: pnpm workspaces + Turborepo
TS end-to-end across `apps/{api,web}` + `packages/{db,shared}`. Boring, mainstream, well-documented (CLAUDE.md §5). pnpm installed via `npm i -g pnpm` (corepack enable failed with EPERM writing to Program Files).

### 2026-06-27 · Data layer: Drizzle ORM + hand-written raw-SQL migrations
Drizzle for typed app queries; DDL + RLS policies + grants are hand-authored SQL in `packages/db/migrations`, applied by a tiny migrator with a `schema_migrations` ledger. Chosen over Prisma because the spine is RLS/ledger-heavy (CLAUDE.md §2). The raw SQL is the source of truth; the Drizzle TS schema is a typed mirror.

### 2026-06-27 · RLS context via per-transaction session GUCs + dedicated app role
The API connects as a non-superuser, non-owner role `app_user` (so RLS binds). Each request runs in a transaction that first sets transaction-local GUCs `app.org_id` / `app.current_party_id` / `app.is_superadmin`; policies read them via `current_setting`. The reusable `withRlsTransaction()` is the single access layer. Alternative (one Postgres role per user) rejected as pooling-hostile.

### 2026-06-27 · Append-only enforced by GRANTs, not just convention
The immutable set (leg, payment, payment_allocation, payment_proof, audit_log) is granted INSERT+SELECT only to `app_user` — no UPDATE/DELETE privilege, so corrections must be reversing entries (CLAUDE.md §3.4). Invoices/rules are mutable-but-no-hard-delete (SELECT/INSERT/UPDATE). Verified by tests.

### 2026-06-27 · A–F dependency tables created early
SCHEMA A–F transitively FK to `ref_entity` (B), `milestone_template(_item)` (H, minimal), `file_object` (G); `audit_log` (G) added too since the access layer needs it. Created now following SCHEMA conventions so the first migration is self-consistent. Other G/H tables deferred.

### 2026-06-27 · Local Postgres on host port 5433
A pre-existing host Postgres occupies 5432 (caused 28P01 auth failures); the Docker container publishes to 5433 and `.env` connection strings use it. Cosmetic/local only — clean machines can use 5432.

### 2026-06-27 · API compiled with tsc (not tsx/esbuild)
NestJS DI relies on `emitDecoratorMetadata`, which esbuild/tsx do not emit. The API is built with `tsc` to ESM and run via `node dist/main.js`. (Module 0 depth has since replaced the header stub with real token auth.)

### 2026-06-27 · Module 0 auth: JWT access+refresh, bcryptjs, otplib TOTP
Access token 30m; refresh is a per-device, hashed-at-rest token with a sliding 10-day expiry re-set on each use; rotation revokes the old token and reuse of a revoked token kills the whole family; logout revokes server-side. Passwords via bcryptjs (pure-JS, portable on Windows); 2FA via otplib TOTP, opt-in. JWT_SECRET must be ≥32 chars or the API refuses to boot, and HS256 is pinned (no alg-confusion / no silent insecure fallback).

### 2026-06-27 · RLS context derives from the signed token, never client input
`AuthGuard` verifies the Bearer access token and sets `req.principal`; `extractRlsContext` builds the RLS GUCs from the principal only — the old `x-org-id/x-party-id/x-superadmin` header path is deleted. Proven by tests: a forged `x-party-id`/`x-superadmin` header has no effect. This closes the impersonation hole the stub left open.

### 2026-06-27 · `is_superadmin` GUC = System SuperAdmin role only (§4.4)
The leg-visibility bypass is granted only when the authenticated principal holds the System SuperAdmin role (computed server-side from `user_role` at login/refresh). Business SuperAdmin deliberately does NOT get it — they get aggregated/settlement views (built later). So no single business seat renders every leg.

### 2026-06-27 · `app_auth_lookup()` SECURITY DEFINER = the sole credential-lookup RLS bypass
Login must read `user_account` by email before any org context exists, which RLS would block. A narrow `SECURITY DEFINER` function (owner-rights, `search_path` pinned, EXECUTE only to `app_user`, returns only auth columns for one email) is the single sanctioned bypass — preferred over a second privileged connection pool. Migration `0003_auth.sql`. The `auth_refresh_token` table is the one mutable security table (UPDATE granted for revoke/rotate); everything else stays append-only.

### 2026-06-27 · Git: commit directly to `main`, no feature branches (user preference)
Module 0 branch `module0-auth-authz-audit` merged (--no-ff) into `main` and deleted. Going forward, work and commit straight to `main`.

### 2026-06-27 · Module 1 fuzzy resolution: normalize + aliases + pg_trgm
"Fuzzy-in / canonical-out" (§7) = three layers: `normalize()` (lowercase, strip non-alphanumerics) collapses case/space/punct variants of the SAME token; explicit `ref_alias` rows cover genuinely different spellings (701 vs ICT701); and `pg_trgm` `similarity()` powers typo-tolerant ranked type-ahead. Exact-normalized hits rank above trigram matches. `unique(org_id, ref_id, normalized)` prevents duplicate aliases on one entity; cross-entity ambiguity is resolved by returning ranked candidates (pick-don't-type), not a hard constraint.

### 2026-06-27 · Module 1 additive schema (0004) — merge redirect + referred-by
`ref_entity` gains `archived_at` + `merged_into_id` so a steward can MERGE a duplicate into a canonical survivor: aliases move to the target, the source's old name is kept as an alias (still resolves), FK refs (`party.university_id`) are repointed, and the source is archived pointing at the survivor. `party` gains `referred_by_party_id` (self-ref) for the directory's "referred-by". Additive nullable columns — not a spine redesign. NOTE: merge's FK-repoint list must grow as ref-consuming tables are added (Module 2: work_item course/assignment refs).

### 2026-06-27 · Module 1 packaged as `refdata/` folder (deny-rule workaround)
`.claude/settings.json` denies `Edit(reference/**)` (meant for the top-level human `/reference/` spreadsheet backup, CLAUDE.md §5), but the glob also blocks an API source dir named `reference/`. Since that settings file is itself deny-listed for edits, the NestJS module lives at `apps/api/src/modules/refdata/` instead; HTTP routes are still `/reference` + `/parties` and the permission module key is still `reference`. Reference data + party directory are one module (Module 1), feature-flagged by `FEATURE_REFERENCE`; the Data Steward role makes the confirm/merge permission delegable to non-owners (§7).

### 2026-06-27 · Module 2 money model: read only from legs (RLS), margins derived
All money (prices, pay, margins) is read from the `leg` chain, which is RLS-enforced (a non-party gets zero rows; only System SuperAdmin bypasses). `deriveMargins` (in `@business-os/shared`) computes margin = inbound − outbound from ONLY the legs the caller can see, and reports a node only when both its sides are visible — so a middle party sees just their own slice and cannot reconstruct the top client price; SuperAdmin sees all nodes. Nothing is stored: no profit/margin column, line amounts computed via `computeLineAmount`. Legs are append-only (insert/select grant only); corrections are reversing legs. Legs are inserted with client-side UUIDs and **no RETURNING**, because an admin building the chain isn't a party to every leg and reading the row back would (correctly) trip the leg SELECT policy.

### 2026-06-27 · Module 2 work_line money + consumer identity redacted to work:approve
`work_line` has only tenant RLS (admins must manage all lines, and they aren't a "party" on consumer rows), so column-level money protection is done in the API: `mapLine` returns the work spec to everyone but withholds `client_rate`/`writer_rate`/`fixed_amount`/computed amount AND the consumer (client) party id unless the caller is System SuperAdmin or holds `work:approve`. This applies on every line-returning path (detail, list, and the `addLine` response). The authoritative, DB-enforced money boundary remains the legs. INTERIM: the `work:approve` gate is coarse; per-field visibility via `permission.scope_json` (SCHEMA §A) is the eventual model (deferred). Full column-level DB security on `work_line` also deferred.

### 2026-06-27 · Module 2 copy fan-out = 1 producer line → N consumer lines
The writer's single "5 copies @ my rate" entry is one producer `work_line` (writer side, `unit_count = N`); the admin's fan-out creates N independent consumer `work_line`s (each its own `consumer_party_id` + `client_rate`/state), each linked back via `source_line_id`. Producer and consumer are never the same row (a line is one side XOR the other; enforced in `addLine`). Legs are created explicitly now; auto-deriving leg amounts from deal terms is Module 3 (`leg.deal_term_id` stays nullable).

### 2026-06-27 · Module 3 effective-dating: half-open windows + supersede (value never mutated)
Rules are date-versioned with half-open windows `[effective_from, effective_to)`. Renegotiation **supersedes**: within one tx, close the prior open version (`UPDATE … SET effective_to = newFrom` — the only sanctioned mutation, allowed by the SELECT/INSERT/UPDATE grant) and INSERT a new version; the prior `value`/`rate` is never changed. So a March job resolved as-of March still hits the March window even after a June renegotiation. `isEffectiveOn`: `effective_from <= asOf AND (effective_to IS NULL OR asOf < effective_to)`. Supersede refuses an already-closed version (no overlapping windows). Resolution filters the window in SQL and the pure ranker re-checks — expired rules can never win.

### 2026-06-27 · Module 3 precedence (most-specific → default)
`resolveDealTerm` scores candidates: specific party-pair (+10) over global null-pair (+0); `applies_to` client (+3) > jobtype (+2) > default (+1); a non-matching client/jobtype rule is excluded; ties break to latest `effective_from`, then latest `created_at`. `resolveCompRule`: party-specific (+10) beats role-level (+1). Pure functions in `@business-os/shared/rules.ts` (unit-tested) so precedence/dating is provable without a DB. `applies_to` follows the SCHEMA text convention (`default` | `client:<uuid>` | `jobtype:<x>`). KNOWN GAP (deferred): jobtype is exact-match free text, so a typo silently no-ops — a canonical-jobtype (ref_entity) tie-in is the eventual fix.

### 2026-06-27 · Module 3 gating + comp_rule provenance (migration 0008)
Deal terms / comp rules are gated by `rules:*` (Admin + System SuperAdmin hold it; Writers don't → money rules don't leak). Party-scoped "own terms" reads (a writer seeing their per-word rate) deferred, consistent with Modules 1/2. SCHEMA §E omitted provenance on `comp_rule` though they are money-defining; migration **0008** additively adds `created_by`/`created_at` (set from the signed-token principal) — a small spine gap closed, not a redesign. Resolution-only this module; auto-applying resolved terms to legs is Module 5. The Module 1 `ReferenceService.merge` was extended here to also repoint `work_item.course_ref_id`/`assignment_type_ref_id` onto the survivor.

### 2026-06-27 · Module 4 web auth: BFF proxy + httpOnly cookies (tokens never in JS)
The browser never holds tokens. It talks only to same-origin Next route handlers (`/api/auth/*`, `/api/proxy/[...path]`); login stores the access+refresh JWTs as **httpOnly, SameSite=Lax** cookies (secure in prod), and the proxy attaches the Bearer from the cookie and refreshes once on 401. A **single-flight** dedupes concurrent refreshes so the rotating refresh token isn't double-spent (which would trip reuse-detection) — per-process (note for multi-instance deploys). The NestJS API remains the real authority (RLS + guards + redaction); the BFF is plumbing. Chosen over client-side token storage (XSS exposure).

### 2026-06-27 · Module 4 CSRF: same-origin required for state-changing proxy calls
Because SameSite=Lax still allows cross-site top-level form POSTs, the proxy rejects unsafe methods (POST/PUT/PATCH/DELETE) unless the request is same-origin (verified via `Sec-Fetch-Site: same-origin`, else a strict `Origin` match). Verified: a cross-site POST → 403. The proxy also rejects traversal path segments (SSRF) and forces `content-type: application/json` to the API. Guards extracted to `src/lib/proxy-guard.ts` and unit-tested.

### 2026-06-27 · Module 4 money-safety: UI renders only what the (redacted) API returns
The DB/API is the money authority (legs RLS + `work_line` redaction). The UI's rule: render money **only when the field is present** — `<Money>`/`formatMoney` return nothing for absent/empty/NaN (never a `0`/`—` that implies a hidden figure), and no client code derives a margin/price or requests an unredacted endpoint. Proven end-to-end: a Writer's job-detail payload lacks `clientRate`/amount, shows only their own leg, no margin. Belt over the server guarantee. SWR + a thin fetch wrapper for reads; a restrained Tailwind primitive set (`src/components/ui.tsx`); shadcn/full visual language still deferred. Deadline/tz-urgency display deferred until a `work_item` deadline field lands with the task board (Module 6).

### 2026-06-28 · Module 5 invoicing + balances: live grouping, derive don't store
Invoices are a **live grouping** of a client's billable lines: a line auto-attaches to the client's open invoice (`ensureOpenInvoice`), lines move between invoices, and an estimate is **superseded by a final** (`supersedes_invoice_id`) with the estimate set `status='void'` but retained (never deleted). All balances are **derived** from `payment_allocation` sums (SCHEMA §I — `invoice_line.paid_amount` is left dormant, never written/read); the only maintained status is `work_item.money_state` (`recomputeMoneyState`: unbilled→invoiced→partial→settled), which is **independent of `work_state`** — the two parallel closes. Payments are events; allocation is the link, supporting partial-within-job and bulk-across-jobs in one call (cap-checked). Money tables (`payment`, `payment_allocation`, `payment_proof`, `charge`) stay INSERT+SELECT only; corrections are **reversing entries** (negative mirrors): a payment reversal carries `reverses_payment_id` and refuses double/over-reversal; `SELECT FOR UPDATE` can't be used to lock (append-only grant withholds UPDATE) so a concurrent-allocation lock (advisory) is deferred.

### 2026-06-28 · Module 5 bidirectional charge ledger (party→business dues)
Legs carry **business→party earnings**; a new append-only `charge` table carries **party→business dues** (platform fee now, AI-check etc. later), itemized as "amount to be paid", settled via `payment_allocation.charge_id`. Each party has a **two-way balance**: `derivePosition` nets earnings-outstanding (legs to them − payouts) against charges-outstanding (their charges − settlements) into a position; a platform fee surfaces as a due on the writer's `/billing/balance/me` (available to any authenticated party; computed under their own RLS). `charge` has **leg-style party-scoped RLS** (a party sees only their own dues). Because an admin isn't a party to a charge, `createCharge` uses client-side id + no RETURNING (like `leg.appendLegs`), and validating/reversing a charge goes through `charge_summary()` (SECURITY DEFINER, EXECUTE to app_user only) so the source party/amount/already-reversed are read server-side, never trusted from the client. Invoices/payments are `billing:*`-gated admin ops; cross-party/business-wide balances are the settlement layer (Phase 2) / SuperAdmin.

### 2026-06-28 · Module 6 expenses: one table, cost-bearer drives attribution
Every cost is one `expense` row (SCHEMA §G): salaries, subscriptions, promo (optional campaign_tag + loose revenue_link_id), losses, events are all categories — no payroll subsystem. The `cost_bearer` (momin/emon/split/writer) is load-bearing for partner economics; a `split` requires a non-empty numeric `cost_bearer_split_json` (validated on create AND update). Expenses are **mutable operational data** (tenant-RLS, full CRUD) — NOT the append-only money ledger — but DELETE is revoked (0012): soft-delete via `archived_at`. Gated by `expenses:*` (admins). Split-amount derivation + revenue-link ROI are deferred (no read-model yet).

### 2026-06-28 · Module 6 tz-aware deadlines + computed urgency
A deadline stores the **absolute instant** (`due_at`, UTC) **plus the IANA zone it was set in** (`due_tz`). The wall-time→instant conversion is a dependency-free, DST-correct Intl helper (`zonedWallToInstant`, offset computed at the target date); an invalid tz → 400 (`isValidTimeZone`). **Urgency / "time left" is absolute and derived, never stored**; the web renders the due moment in the **viewer's browser zone** (and shows the original zone only when it differs — "your time" / "set for <zone>"). Per-user fixed clocks (multi-clock §8) deferred. Tasks are gated by `capture:*` and are an **org-wide shared board** (not party-row-scoped) — honoring `permission.scope_json` rows is a future refinement; `?mine=true` filters to the caller.

### 2026-06-28 · Global date rule: dd/mm/yyyy display + picker-only entry
All user-facing dates render **dd/mm/yyyy** (`formatDate`/`formatDateTime` in `@business-os/shared`, re-exported by the web); existing `toLocaleDateString` usages were retrofitted. Date/time entry is always a **native picker** (`<input type=date|time>`) + a timezone `<Select>` — never a free-text date field. The web depends on `@business-os/shared` (via `transpilePackages`) so the date/money/enums logic lives once. This completes Phase 1 (Modules 0–6).

### 2026-06-28 · Module 5 deferrals S1 + N1 resolved (migration 0013)
**S1 (concurrent allocation):** `payment.service.allocate` now takes a transaction-scoped **`pg_advisory_xact_lock(hashtext(paymentId))`** as its first statement, serializing concurrent allocations on the same payment so the read-then-cap-check-then-insert can't be raced into over-allocation. Earlier the lock seemed to fail with "permission denied for table payment" — that was a **stale `SELECT FOR UPDATE` build** (FOR UPDATE needs UPDATE privilege the append-only payment grant withholds); the advisory lock is a function call needing **no table privilege**, verified working as `app_user`. Covered by a parallel-allocation test (two ×60 vs a 100 payment → exactly one 400, derived paid=60). **N1 (`invoice_line.paid_amount`):** confirmed **derive-only** — removed from the Drizzle mirror (so app code can't reference it; the build proves nothing did) and DB-`COMMENT`ed as deprecated; the physical column is kept for SCHEMA §F fidelity but paid/due are always summed from `payment_allocation` (§I).

### 2026-06-28 · Deal-term auto-pricing wired into the leg engine (capture-first)
Building the money chain no longer requires hand-typing every leg amount. When a leg is appended **without an `amount`**, the engine resolves the deal term on its `(from→to)` relationship **as of the job date** and fills `leg.amount` + `leg.deal_term_id`: `per_word` → value × word_count, `fixed` → value. An **explicit amount overrides** and clears the term link (bespoke price, not "from term X"). Only `per_word`/`fixed` price a leg — the `%` term types (split/commission/referral) divide *margin* and belong to settlement (Phase 2), not to a chain leg. Unpriceable + no amount → 400 (capture stays explicit, never a silent 0). A new read-only **`POST /work/:id/legs/propose`** (`@HttpCode(200)`, gated `work:approve`) returns proposed amounts + `source` (derived/manual/unpriced) **without writing**, so add-a-job can show rule-proposed prices for review/override before committing.

Design choices: (1) **No migration** — `leg.deal_term_id` existed since 0006. (2) The new `PricingService` lives in the **work** module and reuses the **shared pure** `resolveDealTerm` (the one source of precedence/effective-dating truth) while doing its own `deal_term` candidate fetch — so the work module does **not** import RulesModule (keeps the feature-flag seam clean; only a trivial parameterized SQL prefilter is co-located, not the ranking logic). (3) `asOf` defaults to the work_item's `created_at`, so a past job prices on past terms even after a later renegotiation. The two non-negotiables touching legs are preserved: margins stay **derived-from-legs** (`deal_term_id` is provenance, not a stored figure; `deriveMargins`/`getVisibleLegs` untouched) and **leg visibility** holds (auto-pricing inserts with no RETURNING; propose computes from terms and never reads back existing legs — the leg-leak test passes on a fully auto-priced chain). qa-test-writer caught a real defect (propose returned the POST-default `201` for a read-only endpoint) → fixed with `@HttpCode(200)`.

### 2026-06-28 · Billing web screens (Module 5 front end)
Built the invoices / payments+allocation / two-way balance UI on the existing capture-first web stack (BFF proxy + httpOnly cookies, SWR, `ui.tsx` primitives, `EntityPicker`). Money rule reaffirmed end-to-end: the UI renders **only** the figures the redacted API returns — every amount/paid/due/net goes through `<Money>`/`formatMoney` (absent ⇒ render nothing, never a `0`/`—`), and the UI **derives no money** (the pure `lib/billing.ts` helpers — `remainingToAllocate`/`clampAmount`/`netLabel` — only operate on numbers the caller can already see; the allocation "remaining" is a client-side convenience and the server's over-allocation cap is the authority). Actions are permission-gated client-side (billing view/create/edit/approve) over the server's RLS+guards.

Decisions: (1) **Balance = own + admin lookup** — `/balance` shows the viewer's own position via `GET /billing/balance/me` to **any party (incl. writers)**, and `billing:view` holders additionally look up any party via `/billing/balance/:partyId` (with add/reverse-charge). (2) **Allocation is guided by counterparty** — an `in` payment pre-loads the client's open invoice lines (due>0) + the counterparty's outstanding charges; an `out` payment offers the writer aggregate; tick several with partial amounts for the partial-within-job + bulk-across-jobs model. The allocation section requires `billing:view` so an unprivileged viewer never triggers a cross-party invoice/charge fetch (security-reviewer B1). (3) **Reversal state is read from `reverses_payment_id`, never from a (possibly hidden) amount** (B1/S1). (4) **Payment-proof attachment deferred** — no file-upload/object-storage pipeline exists, so nothing can create a `file_object` to attach. (5) Live-grouping capture entry: the **job hub** bills a money-visible consumer line via `POST /invoices/attach-line` (auto-attaches to the client's open invoice). Known gap: no `GET /payments/:id`, so the payment detail page fetches the list and finds by id, and "remaining" is session-local — a tracked backend follow-up.

### 2026-06-28 · Projects / engagements + milestones + milestone templates (DESIGN_SPEC §5)
A project is a **container of child work_items** flagged trackable/billable; **a plain job is the one-child case**, so projects use the **same machinery** as jobs — children are ordinary `work_item`s (no parallel type), and the whole projects module is gated by the existing **`work:*`** permission module and mounts under `FEATURE_WORK` (no new permission module, no new feature flag, no seed migration). The spine tables already existed (0000); migration **0014** only adds `work_item.trackable/billable`, project/milestone provenance columns, and indexes.

Decisions: (1) **Templates are work-gated** (admins manage them) rather than reference-gated — operationally simplest, avoids an admin lacking `reference:*`; they're per-uni/programme lists you **instantiate then extend** (snapshot the items into real milestones on create, append more via `/instantiate` — fluid programmes; template edits never mutate an in-flight project). (2) **Self-contained estimate→actual**: `project.estimate_amount` stores the rough quote; the **actual is DERIVED** at read time = Σ over **billable** children of their consumer-line client amounts (`computeLineAmount`), never stored (§3.3). Both estimate and actual are **money-gated** (System SuperAdmin / `work:approve`); crucially the stored `estimate_amount` is **redacted from the raw project row** for non-money callers too — a reviewer/qa pass caught that returning the project row leaked the estimate past the gated `money` block (§3.2). Firming the client invoice stays in the billing module's estimate→final supersede (module seam kept; projects don't call billing). (3) Milestones reuse the task module's **tz-aware deadline** pattern (absolute instant + IANA zone via `zonedWallToInstant`/`isValidTimeZone`; urgency derived, never stored) and a `pending→in_progress→done` state machine; `complete` on a project is the governance step (work:approve, stamps confirmed_by/at), and re-completing returns 409.

### 2026-06-28 · Settlement layer (partner balance, transfers, split/commission, platform fee) · §4.4, §3
Built the shared Emon↔Momin settlement layer. The §4.4 crux ("see the other's work volume" ≠ "see the other's margins") is enforced at the DB: a new **SECURITY DEFINER** `settlement_legs(a,b)` bypasses leg RLS but returns, per shared job, ONLY the **shared pool** (the downstream-node margin = inter-partner handoff leg − the downstream's outbound legs to non-partners) plus the pair + job date — never the upstream's private client leg. It is **caller-guarded** (`app_is_superadmin() OR app_current_party() in (a,b)`), so a third party or a billing:view admin who isn't one of the two partners gets zero rows. Raw legs never appear in any settlement response; the unchanged `leg_visibility` RLS still blocks direct leg reads.

Money model (logged so it isn't re-derived): **shared pool = downstream-node margin only** — the upstream's private margin (their top leg − the handoff) is never in the pool (matches §3.1's 6000/5000/3000 → pool 2000, Momin's private 1000 invisible to Emon). **One formula for both term types**: for the inter-partner handoff `from=upstream → to=downstream`, the downstream holds the pool and **owes the upstream `value%`** of it, where `value` is the resolved `split_pct` or `commission_pct` term on (upstream→downstream) as-of the job date (split 50 on Momin→Emon → Emon owes Momin 1000; commission 20 on Emon→Momin → Momin owes Emon 20% — "Emon takes 20%, Momin keeps the rest"). All split/net/who-owes-whom is **derived at read time** (pure `deriveSettlement`); only `settlement_transfer` rows (append-only, dated, reversible, party-RLS) and `charge` rows are stored.

**Platform/system fee** = a new `platform_fee` deal-term type (a %); an explicit `POST /settlement/platform-fee` resolves it (via the shared `resolveDealTerm`, effective-dated) and inserts a party→business `charge` = pct × the party's earnings on the job (`party_job_earnings` definer for the base, since the admin isn't a leg-party). Idempotent: a `platform_fee_exists` check + a partial-unique index (migration 0016) backstops the check-then-insert race. Gated by the existing `billing:*` module (both partners are Admins) — no new permission module/seed; `SettlementModule` mounts under FEATURE_BILLING. Reviews fixed a transfer-netting **sign bug** (a settling payment had doubled the balance), the platform-fee term-version resolution (was hand-rolled/oldest), and the idempotency race.

### 2026-06-28 · Per-work outcomes + derived reputation + writer capacity (§8)
Built Module 7 (`outcomes`). ### 2026-06-28 · Credential vault (encrypted at rest, per-item sharing, 2FA-gated reveal) · §8, CLAUDE.md §4
Built Module 8 (`credential_vault`) — a secrets-manager for tool accounts AND client-scoped portal credentials. Each item carries metadata (name, `type` portal/google/github/aws/tool/other, `url`, `client_party_id`) plus a secret bundle {username,password,2FA-recovery,notes} that is **AES-256-GCM encrypted by the app before insert** — the DB only ever holds `secret_iv`/`secret_tag`/`secret_ciphertext`; plaintext never lands in a column, an audit `detail`, or a log. New `EncryptionService` reads `VAULT_ENCRYPTION_KEY` (base64 32 bytes) and hard-fails at boot if missing/short (mirrors the JWT_SECRET check); it's provided only by the vault module, so the key is required only when the vault is enabled.

Decisions: (1) **Per-item ACL enforced in RLS** (`credential_vault_item` visible only to a party with an active `credential_share`, or System SuperAdmin) — a non-holder gets ZERO rows, exactly like legs; revoke (`revoked_at`) is instant. This is "visibility at the DB" (CLAUDE §3.2), not app-enforced. (2) **Holders-only reveal** — admins are not auto-holders; an admin who must reveal grants themselves a share first (audited), so "who could ever see this secret" is always reconstructable. (3) **Mandatory TOTP step-up on reveal**: the caller must have 2FA enrolled and present a valid current code, else 403/401; every reveal (allowed and denied) is audited; the secret is returned only in the HTTPS response body. (4) **Admin management via SECURITY DEFINER** (`vault_manage_list`/`vault_manage_shares`/`vault_revoke_share`, metadata-only, org-filtered), reachable only from `credential_vault:approve` endpoints — the sanctioned RLS-bypass pattern. (5) **Mutable-but-undeletable** (grants select/insert/update; items archive, shares revoke). New `credential_vault` permission module (Admin all; Writer view-own). Reviews fixed: reveal returns 200 (was POST-default 201); `grant` validates the grantee party is in-org. **Follow-up**: `user_account.twofa_secret` is still plaintext — encrypt it with the new EncryptionService (a DB compromise currently exposes the TOTP seed). Needs `FEATURE_CREDENTIAL_VAULT=true` + `VAULT_ENCRYPTION_KEY` in `.env`.

### 2026-06-28 · File pipeline + Knowledge base + Cover sheets (§7/§8, file rule)

The pipeline is core plumbing (FilesModule, always-on) since payment-proofs/briefs will reuse it. **File rule enforced at `POST /files`**: small files stored, large files + video are **links only**, **images recompressed (sharp)**, and the **DB stores only metadata** — `file_object.url` is a server-generated storage key (or an external link), never the bytes, and the key is never returned to clients (stored files download via `GET /files/:id/download`).

(4) The web BFF proxy is JSON-only, so uploads/downloads use dedicated routes (`/api/upload` multipart, `/api/files/[id]` binary-safe stream) that keep tokens server-side.

Security review fixes: the download endpoint serves **stored files only** — a link is opened directly from its metadata URL, never bounced through our origin (**no open redirect**); `file.linked` audit logs only the URL **host** (link URLs can carry tokens); the stored **storage key is masked** (null) in article-attachment responses; non-image stored files are served `attachment` + `X-Content-Type-Options: nosniff` (**no inline stored-XSS**); **publish status is curator-only** (authors can't self-publish). **Known gap (tracked)**: `GET /files/:id/download` is org-scoped but has **no per-file ACL** — fine for org-public knowledge/cover-sheets, but a kind-aware ACL must be added before sensitive kinds (proof/brief/receipt) reuse the pipeline. ### 2026-06-28 · AI/plagiarism check service — a self-contained mini-business (§8)
Built Module 10 (`checks`). A semi-separate unit (2–3 employees) collects files from their OWN WhatsApp customers (often not academic clients — a check stands alone), runs them through tool accounts (AcademyCX), and collects payment. The model: **a `check_channel` = an employee × a WhatsApp account**; the **`check_batch` is the primary capture** — a low-friction tally per channel×day (files_checked / files_paid / amount_collected), optionally linked to a customer party and/or a work_item, with an optional `check_file` per-file detail (the file + AI/plagiarism score). A per-file check is just a batch of one.

Governance (§2): a worker **proposes** a batch (a claim); an admin/QA **confirms** it (**no self-confirm** — confirmer ≠ recorder), and **only confirmed batches feed the P&L and consume credits** — that's the governance's teeth. **Standalone P&L is DERIVED, never stored** (`packages/shared/src/checks.ts`): revenue collected − **allocated account cost** − **worker comp**. The account cost is grounded in **real top-ups** — `check_credit_topup` is append-only (credits + price); cost-per-credit = Σcost/Σcredits (the unrounded ratio is used for the cost calc, rounded once, so per-credit rounding doesn't drift at volume); allocated cost = confirmed files × that. Worker comp = confirmed files × the employee's per-file `comp_rule` (reusing `resolveCompRule`); a P&L `unpricedBatches` flag surfaces files with no comp rule so net isn't silently overstated. AcademyCX thus appears as both **capacity** (credit balance = purchased − consumed, derived) and **cost** — answering "are we making money on checks?". The vault link is an id only (the secret is never read here).

Decisions: (1) **Cost basis = real top-ups** (not a fixed unit cost). (2) **Confirmed batches are immutable** — corrections are a fresh batch, not an in-place edit (the §3.4 ledger ethos); top-ups are append-only. (3) **Visibility**: a worker sees only their own channels' batches; the credit/cost + P&L are **admin-only** (`checks:approve`); `checks` is gated by its own permission module (operational roles a4–a7 get view+create). (4) **Batch own-only is enforced app-tier over tenant-RLS, NOT party-RLS like legs** — batches aren't the structurally-opaque money-leg class, business admins (not System SuperAdmin) legitimately need the whole board, and the sensitive cost/P&L is separately gated; logged here per the §3.2 "visibility at the DB" review note. Needs `FEATURE_CHECKS=true`.

### 2026-06-28 · Referrers — referral income as a claimant leg (§4, §8)
Built Module 11 (`referrers`). Per the owner, **a referral is "another claimant leg, scoped like any other"**: an admin attaches a `leg` from the business (`from_party_id = NULL`) **to a referrer** party, and the existing `leg_visibility` RLS (0001) scopes it so the referrer sees ONLY their own slice — never the client price or the rest of the chain. Payout flows through the **existing earnings model** (legs-to-you in `balance.service` + `payment_allocation`), so "payout to their earnings" needed no new money plumbing. The referrer login (role a9, `referrers:view`) gets only `GET /referrers/me` (own income + the works that generated it, via the `referrer_works` SECURITY DEFINER caller-guarded to `app_current_party()`).

**Reconciliation with the 2026-06-28 auto-pricing decision** ("% term types divide margin and belong to settlement, not a chain leg"): that still governs the **auto-priced client→writer chain** (the chain pricer remains per_word/fixed only). A referral leg is a *deliberate, admin-attached side-leg* whose amount is computed from a `referral_pct` agreement against a base an admin can see — a distinct path, not the chain pricer.

Decisions: (1) **Flexible basis** — the agreement is a `deal_term` (`term_type='referral_pct'`, `from=referrer`) with a new nullable **`basis` ∈ {revenue, margin, fixed}** (value = pct for revenue/margin, = amount for fixed); per-client override via `applies_to=client:<id>`. The agreement yields a **suggestion**; the admin may **override** the final leg amount (intentionally **unbounded** — the owner asked for maximum flexibility; corrections are reversing legs since money is append-only). (2) **Attach model = both** — a client carries a default referrer (`party.referred_by_party_id`) AND an admin can attach/override per job. (3) **One-hop, NO cascade of dividends** (load-bearing owner rule): the beneficiary is the client's *direct* referrer only; we never walk the `referred_by` graph upward (Mujib→Siam→clients ⇒ Siam earns on his, never Mujib), though an admin may explicitly reassign the beneficiary per job. A party can be both client and referrer (Siam); a client can't be their own referrer on their own job. (4) **Admin direct-attach** (no propose→confirm) — reuses the authorized leg path (`referrers:approve`); a referral can only target an **existing `party_type 'referrer'`** (no free-type). (5) **Every referral leg references a `referral_pct` deal_term** (the marker + provenance); when an admin enters a manual amount with no standing agreement, a **one-off fixed term is auto-created** so the leg is still identifiable as a referral. (6) **New `referrers` permission module** (a1+a3 all; a2+a4 view; a9 view). 

Definer-function guards: `referrer_works` is caller-guarded (own slice). `referral_exists` (dup guard) returns a **boolean only** — security review (H1) flagged the original amount-returning `referral_net` as an unguarded money-leak primitive, so it was reduced to existence (mirrors `platform_fee_exists`). `job_money` (revenue + writer cost, the suggestion base) is org-scoped and relies on the `referrers:approve` endpoint gate — the same trust model as `party_job_earnings` (0015); a Writer hitting suggest/attach gets 403 (tested). `client_name` exposed to the referrer is authorized by **§4.5** ("Client identity … Referrer: own referred R"). Needs `FEATURE_REFERRERS=true`.
