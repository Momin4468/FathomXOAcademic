# DECISIONS.md

> Append-only architectural decision log. **Append (never edit/delete) an entry whenever a foundational decision is made** (per CLAUDE.md ¬ß1). Newest at the bottom. Format: date ¬∑ decision ¬∑ why. This is the "why things are the way they are" record ‚Äî read before changing anything structural.

---

### 2026-06 ¬∑ Stack: PostgreSQL + TypeScript + NestJS + Next.js + embedded Metabase
Postgres chosen over MySQL for native row-level security, JSONB (custom fields), and ledger/analytics strength ‚Äî all core requirements. TS end-to-end for one-language maintainability. Modular monolith.

### 2026-06 ¬∑ Profit is derived from legs, never stored
Margins/splits/commission computed at read time from the leg chain so they stay correct as date-versioned deal terms change. No profit/margin column anywhere.

### 2026-06 ¬∑ Visibility enforced at the database (RLS), not just UI
A user may read a money leg only if SuperAdmin or their party is on it. Enforced by Postgres RLS so a UI bug cannot leak a price.

### 2026-06 ¬∑ Personal Finance is a separable service, linked not merged
Own identity/data/subscription; one-way income API from business payouts; business cannot read into it; deactivating a brokerage account does not disable it.

### 2026-06 ¬∑ Two parallel closes per job
Work-state (draft‚Üípending‚Üíconfirmed‚Üídelivered) and money-state (unbilled‚Üíinvoiced‚Üípartial‚Üísettled) are independent.

### 2026-06-27 ¬∑ Data layer: Drizzle ORM + hand-written raw-SQL migrations
Drizzle for typed app queries; DDL + RLS policies + grants are hand-authored SQL in `packages/db/migrations`, applied by a tiny migrator with a `schema_migrations` ledger. ### 2026-06-27 ¬∑ RLS context via per-transaction session GUCs + dedicated app role
The API connects as a non-superuser, non-owner role `app_user` (so RLS binds). Each request runs in a transaction that first sets transaction-local GUCs `app.org_id` / `app.current_party_id` / `app.is_superadmin`; policies read them via `current_setting`. The reusable `withRlsTransaction()` is the single access layer. Alternative (one Postgres role per user) rejected as pooling-hostile.

### 2026-06-27 ¬∑ Append-only enforced by GRANTs, not just convention
The immutable set (leg, payment, payment_allocation, payment_proof, audit_log) is granted INSERT+SELECT only to `app_user` ‚Äî no UPDATE/DELETE privilege, so corrections must be reversing entries (CLAUDE.md ¬ß3.4). Invoices/rules are mutable-but-no-hard-delete (SELECT/INSERT/UPDATE). Verified by tests.

### 2026-06-27 ¬∑ Local Postgres on host port 5433
A pre-existing host Postgres occupies 5432 (caused 28P01 auth failures); the Docker container publishes to 5433 and `.env` connection strings use it. Cosmetic/local only ‚Äî clean machines can use 5432.

### 2026-06-27 ¬∑ API compiled with tsc (not tsx/esbuild)
NestJS DI relies on `emitDecoratorMetadata`, which esbuild/tsx do not emit. The API is built with `tsc` to ESM and run via `node dist/main.js`. (Module 0 depth has since replaced the header stub with real token auth.)

### 2026-06-27 ¬∑ Module 0 auth: JWT access+refresh, bcryptjs, otplib TOTP
Access token 30m; refresh is a per-device, hashed-at-rest token with a sliding 10-day expiry re-set on each use; rotation revokes the old token and reuse of a revoked token kills the whole family; logout revokes server-side. Passwords via bcryptjs (pure-JS, portable on Windows); 2FA via otplib TOTP, opt-in. JWT_SECRET must be ‚â•32 chars or the API refuses to boot, and HS256 is pinned (no alg-confusion / no silent insecure fallback).

### 2026-06-27 ¬∑ RLS context derives from the signed token, never client input
`AuthGuard` verifies the Bearer access token and sets `req.principal`; `extractRlsContext` builds the RLS GUCs from the principal only ‚Äî the old `x-org-id/x-party-id/x-superadmin` header path is deleted. Proven by tests: a forged `x-party-id`/`x-superadmin` header has no effect. This closes the impersonation hole the stub left open.

### 2026-06-27 ¬∑ `is_superadmin` GUC = System SuperAdmin role only (¬ß4.4)
The leg-visibility bypass is granted only when the authenticated principal holds the System SuperAdmin role (computed server-side from `user_role` at login/refresh). Business SuperAdmin deliberately does NOT get it ‚Äî they get aggregated/settlement views (built later). So no single business seat renders every leg.

### 2026-06-27 ¬∑ `app_auth_lookup()` SECURITY DEFINER = the sole credential-lookup RLS bypass
Login must read `user_account` by email before any org context exists, which RLS would block. A narrow `SECURITY DEFINER` function (owner-rights, `search_path` pinned, EXECUTE only to `app_user`, returns only auth columns for one email) is the single sanctioned bypass ‚Äî preferred over a second privileged connection pool. Migration `0003_auth.sql`. The `auth_refresh_token` table is the one mutable security table (UPDATE granted for revoke/rotate); everything else stays append-only.

### 2026-06-27 ¬∑ Module 1 fuzzy resolution: normalize + aliases + pg_trgm
"Fuzzy-in / canonical-out" (¬ß7) = three layers: `normalize()` (lowercase, strip non-alphanumerics) collapses case/space/punct variants of the SAME token; explicit `ref_alias` rows cover genuinely different spellings (701 vs ICT701); and `pg_trgm` `similarity()` powers typo-tolerant ranked type-ahead. Exact-normalized hits rank above trigram matches. `unique(org_id, ref_id, normalized)` prevents duplicate aliases on one entity; cross-entity ambiguity is resolved by returning ranked candidates (pick-don't-type), not a hard constraint.

### 2026-06-27 ¬∑ Module 2 money model: read only from legs (RLS), margins derived
All money (prices, pay, margins) is read from the `leg` chain, which is RLS-enforced (a non-party gets zero rows; only System SuperAdmin bypasses). `deriveMargins` (in `@business-os/shared`) computes margin = inbound ‚àí outbound from ONLY the legs the caller can see, and reports a node only when both its sides are visible ‚Äî so a middle party sees just their own slice and cannot reconstruct the top client price; SuperAdmin sees all nodes. Nothing is stored: no profit/margin column, line amounts computed via `computeLineAmount`. Legs are append-only (insert/select grant only); corrections are reversing legs. Legs are inserted with client-side UUIDs and **no RETURNING**, because an admin building the chain isn't a party to every leg and reading the row back would (correctly) trip the leg SELECT policy.

### 2026-06-27 ¬∑ Module 2 work_line money + consumer identity redacted to work:approve
`work_line` has only tenant RLS (admins must manage all lines, and they aren't a "party" on consumer rows), so column-level money protection is done in the API: `mapLine` returns the work spec to everyone but withholds `client_rate`/`writer_rate`/`fixed_amount`/computed amount AND the consumer (client) party id unless the caller is System SuperAdmin or holds `work:approve`. This applies on every line-returning path (detail, list, and the `addLine` response). The authoritative, DB-enforced money boundary remains the legs. INTERIM: the `work:approve` gate is coarse; per-field visibility via `permission.scope_json` (SCHEMA ¬ßA) is the eventual model (deferred). Full column-level DB security on `work_line` also deferred.

### 2026-06-27 ¬∑ Module 2 copy fan-out = 1 producer line ‚Üí N consumer lines
The writer's single "5 copies @ my rate" entry is one producer `work_line` (writer side, `unit_count = N`); the admin's fan-out creates N independent consumer `work_line`s (each its own `consumer_party_id` + `client_rate`/state), each linked back via `source_line_id`. Producer and consumer are never the same row (a line is one side XOR the other; enforced in `addLine`). Legs are created explicitly now; auto-deriving leg amounts from deal terms is Module 3 (`leg.deal_term_id` stays nullable).

### 2026-06-27 ¬∑ Module 3 effective-dating: half-open windows + supersede (value never mutated)
Rules are date-versioned with half-open windows `[effective_from, effective_to)`. Renegotiation **supersedes**: within one tx, close the prior open version (`UPDATE ‚Ä¶ SET effective_to = newFrom` ‚Äî the only sanctioned mutation, allowed by the SELECT/INSERT/UPDATE grant) and INSERT a new version; the prior `value`/`rate` is never changed. So a March job resolved as-of March still hits the March window even after a June renegotiation. `isEffectiveOn`: `effective_from <= asOf AND (effective_to IS NULL OR asOf < effective_to)`. Supersede refuses an already-closed version (no overlapping windows). Resolution filters the window in SQL and the pure ranker re-checks ‚Äî expired rules can never win.

### 2026-06-27 ¬∑ Module 3 precedence (most-specific ‚Üí default)
`resolveDealTerm` scores candidates: specific party-pair (+10) over global null-pair (+0); `applies_to` client (+3) > jobtype (+2) > default (+1); a non-matching client/jobtype rule is excluded; ties break to latest `effective_from`, then latest `created_at`. `resolveCompRule`: party-specific (+10) beats role-level (+1). Pure functions in `@business-os/shared/rules.ts` (unit-tested) so precedence/dating is provable without a DB. `applies_to` follows the SCHEMA text convention (`default` | `client:<uuid>` | `jobtype:<x>`). KNOWN GAP (deferred): jobtype is exact-match free text, so a typo silently no-ops ‚Äî a canonical-jobtype (ref_entity) tie-in is the eventual fix.

### 2026-06-27 ¬∑ Module 3 gating + comp_rule provenance (migration 0008)
Deal terms / comp rules are gated by `rules:*` (Admin + System SuperAdmin hold it; Writers don't ‚Üí money rules don't leak). Party-scoped "own terms" reads (a writer seeing their per-word rate) deferred, consistent with Modules 1/2. SCHEMA ¬ßE omitted provenance on `comp_rule` though they are money-defining; migration **0008** additively adds `created_by`/`created_at` (set from the signed-token principal) ‚Äî a small spine gap closed, not a redesign. Resolution-only this module; auto-applying resolved terms to legs is Module 5. The Module 1 `ReferenceService.merge` was extended here to also repoint `work_item.course_ref_id`/`assignment_type_ref_id` onto the survivor.

### 2026-06-27 ¬∑ Module 4 web auth: BFF proxy + httpOnly cookies (tokens never in JS)
The browser never holds tokens. It talks only to same-origin Next route handlers (`/api/auth/*`, `/api/proxy/[...path]`); login stores the access+refresh JWTs as **httpOnly, SameSite=Lax** cookies (secure in prod), and the proxy attaches the Bearer from the cookie and refreshes once on 401. A **single-flight** dedupes concurrent refreshes so the rotating refresh token isn't double-spent (which would trip reuse-detection) ‚Äî per-process (note for multi-instance deploys). The NestJS API remains the real authority (RLS + guards + redaction); the BFF is plumbing. Chosen over client-side token storage (XSS exposure).

### 2026-06-27 ¬∑ Module 4 CSRF: same-origin required for state-changing proxy calls
Because SameSite=Lax still allows cross-site top-level form POSTs, the proxy rejects unsafe methods (POST/PUT/PATCH/DELETE) unless the request is same-origin (verified via `Sec-Fetch-Site: same-origin`, else a strict `Origin` match). Verified: a cross-site POST ‚Üí 403. The proxy also rejects traversal path segments (SSRF) and forces `content-type: application/json` to the API. Guards extracted to `src/lib/proxy-guard.ts` and unit-tested.

### 2026-06-27 ¬∑ Module 4 money-safety: UI renders only what the (redacted) API returns
The DB/API is the money authority (legs RLS + `work_line` redaction). The UI's rule: render money **only when the field is present** ‚Äî `<Money>`/`formatMoney` return nothing for absent/empty/NaN (never a `0`/`‚Äî` that implies a hidden figure), and no client code derives a margin/price or requests an unredacted endpoint. Proven end-to-end: a Writer's job-detail payload lacks `clientRate`/amount, shows only their own leg, no margin. Belt over the server guarantee. SWR + a thin fetch wrapper for reads; a restrained Tailwind primitive set (`src/components/ui.tsx`); shadcn/full visual language still deferred. Deadline/tz-urgency display deferred until a `work_item` deadline field lands with the task board (Module 6).

### 2026-06-28 ¬∑ Module 5 invoicing + balances: live grouping, derive don't store
Invoices are a **live grouping** of a client's billable lines: a line auto-attaches to the client's open invoice (`ensureOpenInvoice`), lines move between invoices, and an estimate is **superseded by a final** (`supersedes_invoice_id`) with the estimate set `status='void'` but retained (never deleted). All balances are **derived** from `payment_allocation` sums (SCHEMA ¬ßI ‚Äî `invoice_line.paid_amount` is left dormant, never written/read); the only maintained status is `work_item.money_state` (`recomputeMoneyState`: unbilled‚Üíinvoiced‚Üípartial‚Üísettled), which is **independent of `work_state`** ‚Äî the two parallel closes. Payments are events; allocation is the link, supporting partial-within-job and bulk-across-jobs in one call (cap-checked). Money tables (`payment`, `payment_allocation`, `payment_proof`, `charge`) stay INSERT+SELECT only; corrections are **reversing entries** (negative mirrors): a payment reversal carries `reverses_payment_id` and refuses double/over-reversal; `SELECT FOR UPDATE` can't be used to lock (append-only grant withholds UPDATE) so a concurrent-allocation lock (advisory) is deferred.

### 2026-06-28 ¬∑ Module 5 bidirectional charge ledger (party‚Üíbusiness dues)
Legs carry **business‚Üíparty earnings**; a new append-only `charge` table carries **party‚Üíbusiness dues** (platform fee now, AI-check etc. later), itemized as "amount to be paid", settled via `payment_allocation.charge_id`. Each party has a **two-way balance**: `derivePosition` nets earnings-outstanding (legs to them ‚àí payouts) against charges-outstanding (their charges ‚àí settlements) into a position; a platform fee surfaces as a due on the writer's `/billing/balance/me` (available to any authenticated party; computed under their own RLS). `charge` has **leg-style party-scoped RLS** (a party sees only their own dues). Because an admin isn't a party to a charge, `createCharge` uses client-side id + no RETURNING (like `leg.appendLegs`), and validating/reversing a charge goes through `charge_summary()` (SECURITY DEFINER, EXECUTE to app_user only) so the source party/amount/already-reversed are read server-side, never trusted from the client. Invoices/payments are `billing:*`-gated admin ops; cross-party/business-wide balances are the settlement layer (Phase 2) / SuperAdmin.

### 2026-06-28 ¬∑ Module 6 expenses: one table, cost-bearer drives attribution
Every cost is one `expense` row (SCHEMA ¬ßG): salaries, subscriptions, promo (optional campaign_tag + loose revenue_link_id), losses, events are all categories ‚Äî no payroll subsystem. The `cost_bearer` (momin/emon/split/writer) is load-bearing for partner economics; a `split` requires a non-empty numeric `cost_bearer_split_json` (validated on create AND update). Expenses are **mutable operational data** (tenant-RLS, full CRUD) ‚Äî NOT the append-only money ledger ‚Äî but DELETE is revoked (0012): soft-delete via `archived_at`. Gated by `expenses:*` (admins). Split-amount derivation + revenue-link ROI are deferred (no read-model yet).

### 2026-06-28 ¬∑ Module 6 tz-aware deadlines + computed urgency
A deadline stores the **absolute instant** (`due_at`, UTC) **plus the IANA zone it was set in** (`due_tz`). The wall-time‚Üíinstant conversion is a dependency-free, DST-correct Intl helper (`zonedWallToInstant`, offset computed at the target date); an invalid tz ‚Üí 400 (`isValidTimeZone`). **Urgency / "time left" is absolute and derived, never stored**; the web renders the due moment in the **viewer's browser zone** (and shows the original zone only when it differs ‚Äî "your time" / "set for <zone>"). Per-user fixed clocks (multi-clock ¬ß8) deferred. Tasks are gated by `capture:*` and are an **org-wide shared board** (not party-row-scoped) ‚Äî honoring `permission.scope_json` rows is a future refinement; `?mine=true` filters to the caller.

### 2026-06-28 ¬∑ Global date rule: dd/mm/yyyy display + picker-only entry
All user-facing dates render **dd/mm/yyyy** (`formatDate`/`formatDateTime` in `@business-os/shared`, re-exported by the web); existing `toLocaleDateString` usages were retrofitted. Date/time entry is always a **native picker** (`<input type=date|time>`) + a timezone `<Select>` ‚Äî never a free-text date field. The web depends on `@business-os/shared` (via `transpilePackages`) so the date/money/enums logic lives once. This completes Phase 1 (Modules 0‚Äì6).

### 2026-06-28 ¬∑ Module 5 deferrals S1 + N1 resolved (migration 0013)
**S1 (concurrent allocation):** `payment.service.allocate` now takes a transaction-scoped **`pg_advisory_xact_lock(hashtext(paymentId))`** as its first statement, serializing concurrent allocations on the same payment so the read-then-cap-check-then-insert can't be raced into over-allocation. Earlier the lock seemed to fail with "permission denied for table payment" ‚Äî that was a **stale `SELECT FOR UPDATE` build** (FOR UPDATE needs UPDATE privilege the append-only payment grant withholds); the advisory lock is a function call needing **no table privilege**, verified working as `app_user`. Covered by a parallel-allocation test (two √ó60 vs a 100 payment ‚Üí exactly one 400, derived paid=60). **N1 (`invoice_line.paid_amount`):** confirmed **derive-only** ‚Äî removed from the Drizzle mirror (so app code can't reference it; the build proves nothing did) and DB-`COMMENT`ed as deprecated; the physical column is kept for SCHEMA ¬ßF fidelity but paid/due are always summed from `payment_allocation` (¬ßI).

### 2026-06-28 ¬∑ Deal-term auto-pricing wired into the leg engine (capture-first)
Building the money chain no longer requires hand-typing every leg amount. When a leg is appended **without an `amount`**, the engine resolves the deal term on its `(from‚Üíto)` relationship **as of the job date** and fills `leg.amount` + `leg.deal_term_id`: `per_word` ‚Üí value √ó word_count, `fixed` ‚Üí value. An **explicit amount overrides** and clears the term link (bespoke price, not "from term X"). Only `per_word`/`fixed` price a leg ‚Äî the `%` term types (split/commission/referral) divide *margin* and belong to settlement (Phase 2), not to a chain leg. Unpriceable + no amount ‚Üí 400 (capture stays explicit, never a silent 0). A new read-only **`POST /work/:id/legs/propose`** (`@HttpCode(200)`, gated `work:approve`) returns proposed amounts + `source` (derived/manual/unpriced) **without writing**, so add-a-job can show rule-proposed prices for review/override before committing.

Design choices: (1) **No migration** ‚Äî `leg.deal_term_id` existed since 0006. (2) The new `PricingService` lives in the **work** module and reuses the **shared pure** `resolveDealTerm` (the one source of precedence/effective-dating truth) while doing its own `deal_term` candidate fetch ‚Äî so the work module does **not** import RulesModule (keeps the feature-flag seam clean; only a trivial parameterized SQL prefilter is co-located, not the ranking logic). (3) `asOf` defaults to the work_item's `created_at`, so a past job prices on past terms even after a later renegotiation. The two non-negotiables touching legs are preserved: margins stay **derived-from-legs** (`deal_term_id` is provenance, not a stored figure; `deriveMargins`/`getVisibleLegs` untouched) and **leg visibility** holds (auto-pricing inserts with no RETURNING; propose computes from terms and never reads back existing legs ‚Äî the leg-leak test passes on a fully auto-priced chain). qa-test-writer caught a real defect (propose returned the POST-default `201` for a read-only endpoint) ‚Üí fixed with `@HttpCode(200)`.

### 2026-06-28 ¬∑ Billing web screens (Module 5 front end)
Built the invoices / payments+allocation / two-way balance UI on the existing capture-first web stack (BFF proxy + httpOnly cookies, SWR, `ui.tsx` primitives, `EntityPicker`). Money rule reaffirmed end-to-end: the UI renders **only** the figures the redacted API returns ‚Äî every amount/paid/due/net goes through `<Money>`/`formatMoney` (absent ‚áí render nothing, never a `0`/`‚Äî`), and the UI **derives no money** (the pure `lib/billing.ts` helpers ‚Äî `remainingToAllocate`/`clampAmount`/`netLabel` ‚Äî only operate on numbers the caller can already see; the allocation "remaining" is a client-side convenience and the server's over-allocation cap is the authority). Actions are permission-gated client-side (billing view/create/edit/approve) over the server's RLS+guards.

Decisions: (1) **Balance = own + admin lookup** ‚Äî `/balance` shows the viewer's own position via `GET /billing/balance/me` to **any party (incl. writers)**, and `billing:view` holders additionally look up any party via `/billing/balance/:partyId` (with add/reverse-charge). (2) **Allocation is guided by counterparty** ‚Äî an `in` payment pre-loads the client's open invoice lines (due>0) + the counterparty's outstanding charges; an `out` payment offers the writer aggregate; tick several with partial amounts for the partial-within-job + bulk-across-jobs model. The allocation section requires `billing:view` so an unprivileged viewer never triggers a cross-party invoice/charge fetch (security-reviewer B1). (3) **Reversal state is read from `reverses_payment_id`, never from a (possibly hidden) amount** (B1/S1). (4) **Payment-proof attachment deferred** ‚Äî no file-upload/object-storage pipeline exists, so nothing can create a `file_object` to attach. (5) Live-grouping capture entry: the **job hub** bills a money-visible consumer line via `POST /invoices/attach-line` (auto-attaches to the client's open invoice). Known gap: no `GET /payments/:id`, so the payment detail page fetches the list and finds by id, and "remaining" is session-local ‚Äî a tracked backend follow-up.

### 2026-06-28 ¬∑ Projects / engagements + milestones + milestone templates (DESIGN_SPEC ¬ß5)
A project is a **container of child work_items** flagged trackable/billable; **a plain job is the one-child case**, so projects use the **same machinery** as jobs ‚Äî children are ordinary `work_item`s (no parallel type), and the whole projects module is gated by the existing **`work:*`** permission module and mounts under `FEATURE_WORK` (no new permission module, no new feature flag, no seed migration). The spine tables already existed (0000); migration **0014** only adds `work_item.trackable/billable`, project/milestone provenance columns, and indexes.

Decisions: (1) **Templates are work-gated** (admins manage them) rather than reference-gated ‚Äî operationally simplest, avoids an admin lacking `reference:*`; they're per-uni/programme lists you **instantiate then extend** (snapshot the items into real milestones on create, append more via `/instantiate` ‚Äî fluid programmes; template edits never mutate an in-flight project). (2) **Self-contained estimate‚Üíactual**: `project.estimate_amount` stores the rough quote; the **actual is DERIVED** at read time = Œ£ over **billable** children of their consumer-line client amounts (`computeLineAmount`), never stored (¬ß3.3). Both estimate and actual are **money-gated** (System SuperAdmin / `work:approve`); crucially the stored `estimate_amount` is **redacted from the raw project row** for non-money callers too ‚Äî a reviewer/qa pass caught that returning the project row leaked the estimate past the gated `money` block (¬ß3.2). Firming the client invoice stays in the billing module's estimate‚Üífinal supersede (module seam kept; projects don't call billing). (3) Milestones reuse the task module's **tz-aware deadline** pattern (absolute instant + IANA zone via `zonedWallToInstant`/`isValidTimeZone`; urgency derived, never stored) and a `pending‚Üíin_progress‚Üídone` state machine; `complete` on a project is the governance step (work:approve, stamps confirmed_by/at), and re-completing returns 409.

### 2026-06-28 ¬∑ Settlement layer (partner balance, transfers, split/commission, platform fee) ¬∑ ¬ß4.4, ¬ß3
Built the shared Emon‚ÜîMomin settlement layer. The ¬ß4.4 crux ("see the other's work volume" ‚âÝ "see the other's margins") is enforced at the DB: a new **SECURITY DEFINER** `settlement_legs(a,b)` bypasses leg RLS but returns, per shared job, ONLY the **shared pool** (the downstream-node margin = inter-partner handoff leg ‚àí the downstream's outbound legs to non-partners) plus the pair + job date ‚Äî never the upstream's private client leg. It is **caller-guarded** (`app_is_superadmin() OR app_current_party() in (a,b)`), so a third party or a billing:view admin who isn't one of the two partners gets zero rows. Raw legs never appear in any settlement response; the unchanged `leg_visibility` RLS still blocks direct leg reads.

### 2026-07-10 ¬∑ PF investment "latest-wins valuation" ‚Äî how a contribution after a mark behaves (clarification of the note above)
Pinning the exact edge the "latest-wins valuation" rule raises, so it never needs re-deriving from the code. **Scenario:** principal 1000 ‚Üí valuation 1200 ‚Üí contribution 300 (no later valuation) ‚Üí read. **Answer (intended):** `currentValue = 1500`, `unrealizedPl = +200`. **Why:** a valuation is a mark *as of its date*, not a permanent absolute ‚Äî cash added or removed AFTER the mark adjusts current value 1:1 (money added is worth its face until you re-mark), so `currentValue = latestValuation + Œ£(contribution ‚àí withdrawal) recorded after the mark`, and `costBasis = principal + Œ£(contribution ‚àí withdrawal)`; the two rise together, leaving `unrealizedPl` **unchanged by a contribution**. Only a valuation moves P/L. This deliberately corrects the naive first cut (`currentValue = latest valuation, full stop`), which showed a **phantom loss** equal to any post-mark contribution (adding 300 made P/L look 300 *worse*) ‚Äî misleading for a net-worth tool where adding money is not a gain or loss. ### 2026-07-11 ¬∑ Person ‚âÝ User ‚âÝ Profile ‚Äî three distinct people-management surfaces (`978516a`)
The user asked to "think how all people management can be shown" and to record different fields for different person types. Decided to make the three planes explicit rather than conflate them:
- **Person (party)** = a business entity in the directory. Multi-hat (Khalid sources AND writes), so the `/people/[id]` record renders a facet PER hat instead of one generic form ‚Äî writer/employee show a work log + earnings, partner/referrer show sourced jobs + share, client links to the dedicated Client 360, vendor links to claims. - **Money on the person record stays leg-derived + RLS-scoped** (same opacity rule as the Work board) ‚Äî the register/earnings show only what the caller may see, never a definer read.
